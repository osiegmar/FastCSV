Are you considering contributing to this project?
That's great; help is always welcome — and contributing is about much more than writing code.

:::note
FastCSV does not accept pull requests.
Maintaining a small, focused, high-quality library means every change has to fit the project's
[goals](/architecture/goals/) and [architecture](/architecture/architecture/),
and meet strict requirements for performance, test coverage, and API design.
In practice, very few drive-by PRs clear that bar, and reviewing them takes time away from moving the project forward.

Please use the channels below instead.
:::

## Reporting a bug

Found something that looks like a bug?
[Open a bug report](https://github.com/osiegmar/FastCSV/issues/new?template=bug_report.yml).
The template will ask for everything needed to reproduce and diagnose it —
the more of those fields you can fill in concretely, the faster a fix can land.

## Suggesting an idea

For concrete feature requests, API changes, or behavior tweaks,
[open a feature request](https://github.com/osiegmar/FastCSV/issues/new?template=feature_request.yml).
The template asks you to describe the problem, the solution you'd like, and alternatives you've considered.

If you're not sure yet whether your idea fits — or you just want to think out loud about a use case —
a [discussion](https://github.com/osiegmar/FastCSV/discussions) is the better starting point.
FastCSV deliberately stays small, fast, and dependency-free,
so many reasonable-sounding features turn out to be out of scope.

## Improving the documentation

Documentation issues — unclear wording, missing examples, broken links, outdated information — are very welcome.
[Open a documentation issue](https://github.com/osiegmar/FastCSV/issues/new?template=documentation.yml)
describing what you'd like to see changed and where.

## Helping other users

Answering questions in [discussions](https://github.com/osiegmar/FastCSV/discussions)
is one of the most valuable ways to contribute.
If you've solved a tricky problem with FastCSV, sharing how you did it helps the next person who hits the same wall.

## Sharing how you use FastCSV

If FastCSV is part of something you've built — an open-source project, a blog post, a benchmark, a talk —
I'd love to hear about it in the [discussions](https://github.com/osiegmar/FastCSV/discussions).
Real-world usage shapes the project's direction.

## Spreading the word

Starring the [repository](https://github.com/osiegmar/FastCSV), recommending FastCSV where it fits,
or mentioning it to a colleague who's wrestling with CSV all help the project stay healthy.
